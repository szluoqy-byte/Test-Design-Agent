## 3. 测试场景详情

### SC-001 普通用户提交实物商品订单

测试场景定义业务情境和验证范围。测试点只描述该场景下的验证目标，不填写测试设计方法、具体测试数据取值、测试步骤或完整预期结果。

#### 3.1 设计约束

设计约束用于描述该场景下生成可执行测试用例所需的执行条件、判定规则和观察方式，供该场景下的全部测试点复用。无法确认的约束应写入“待确认信息”。

##### 必填约束

| 约束项 | 内容 |
|---|---|
| 执行入口 | 首页 > 商品详情页 > 立即购买 > 订单确认页 |
| 执行角色 | 普通用户 user_a |
| 数据条件 | 用户已登录并具备下单权限；用户已配置有效收货地址；系统存在上架实物商品；商品库存可按测试点分别设置为充足、不足或不可售 |
| 判定规则 | 库存充足、商品上架且收货地址有效时允许提交订单；库存不足、商品下架或收货地址缺失时阻止提交订单 |
| 观察点 | 页面提示、订单记录、订单状态、商品库存状态 |

##### 可选约束

| 约束项 | 内容 |
|---|---|
| 关键操作 | 点击“立即购买”；在订单确认页点击“提交订单” |
| 状态约束 | 提交成功后订单状态为待支付；提交失败时不得生成订单 |
| 禁止行为 | 库存不足、商品下架或地址缺失时不得创建订单；提交失败时不得扣减库存 |
| 外部依赖 | 库存服务可用；支付服务不在本次范围 |
| 异常处理规则 | 库存不足时提示“库存不足”；收货地址缺失时提示用户维护收货地址 |
| 排除范围 | 不覆盖支付流程；不覆盖优惠券和运费计算 |
| 补充说明 | 提交订单无二次确认弹窗 |

#### 3.2 测试点

| 测试点 ID | 测试点 | 类型 | 级别 | 风险/备注 |
|---|---|---|---|---|
| TP-001 | 库存充足、商品上架且收货地址有效时允许提交订单 | 功能正确性 | Level 0 | 核心主链路 |
| TP-002 | 库存不足时阻止提交订单 | 异常处理 | Level 1 | 防止超卖 |
| TP-003 | 收货地址缺失时阻止提交订单 | 异常处理 | Level 1 | 防止无效订单 |
| TP-004 | 商品下架后阻止提交订单 | 异常处理 | Level 1 | 防止不可售商品下单 |

测试点颗粒度要求：

- 应表达一个明确的规则、风险、路径或条件。
- 不填写测试设计方法，例如边界值、判定表、状态转换。
- 不展开具体边界值、组合矩阵或状态迁移矩阵。
- 不填写测试步骤或完整预期结果。
- 如果一个测试点同时包含多个独立规则，应拆成多个测试点。
