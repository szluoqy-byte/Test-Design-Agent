# 可执行性门禁

## 通过条件

- 前置步骤足以准备账号、数据、环境和状态。
- 测试步骤按执行顺序排列。
- 每一步有明确操作对象。
- 前置步骤、测试步骤和预期结果均使用有序列表，且每条步骤是完整句子。
- GUI 场景测试用例的第 1 条测试步骤描述登录带环境类型标签的环境，例如 ADC、FI、Web 或管理后台，保证每条用例状态独立。
- GUI 场景测试用例的测试步骤包含完整入口操作链路，能够从已登录或约定起始页面进入目标页面、页签或功能区。
- GUI 场景测试用例中的点击、输入、选择、清空、打开、切换、关闭等动作包含可定位的页面、控件或区域对象，优先使用【对象】标记。
- 缺少具体入口、按钮、账号或数据时，已进入待确认问题或使用中性可替换表达。

## 失败信号

- “验证功能正常”“检查结果正确”“执行测试”“进行测试”“流程正常”等空泛步骤。
- 关键前置条件缺失。
- 操作对象不明确。
- 前置步骤、测试步骤或预期结果缺少有序列表，或步骤/预期不是完整句子。
- GUI 场景测试用例第 1 条测试步骤不是登录带环境类型标签的环境，只写“登录环境/系统”，或依赖上一条用例的登录态、页面状态。
- GUI 场景测试用例只在前置步骤中写“已进入目标页面”，测试步骤未写出点击菜单、页面、页签或触发入口的完整链路。
- context-pack 已提供补全入口路径，但测试步骤仍使用“进入某菜单”“打开目标页面”等泛化入口表达。
- GUI 场景测试用例包含“查看页面”“检查列表”“执行查询”“进行操作”等不可直接自动化定位的泛化步骤。
- 步骤无法由测试人员实际执行。
