#### 3.1 设计约束

<!--
填写说明：

##### 必填约束

| 约束项 | 内容 |
|---|---|
| 执行入口 | 例如：首页 > 商品详情页 > 立即购买 > 订单确认页 |
| 执行角色 | 例如：普通用户 user_a；管理员 admin_a；无权限用户 user_b |
| 数据条件 | 例如：库存大于 1 的上架商品；库存为 0 的上架商品；有效收货地址 |
| 判定规则 | 例如：库存大于 0 时允许提交；库存等于 0 时不允许提交 |
| 观察点 | 例如：页面提示、订单记录、库存记录、接口响应、日志/审计 |

##### 可选约束

| 约束项 | 内容 |
|---|---|
| 关键操作 | 例如：点击“立即购买”；点击“提交订单”；弹窗点击“确认取消” |
| 状态约束 | 例如：提交成功后订单为待支付；库存不足时不得创建订单 |
| 禁止行为 | 例如：库存不足时不得生成订单；无权限用户不得查看他人数据 |
| 外部依赖 | 例如：库存服务可用；定时任务已开启；支付服务不在本次范围 |
| 异常处理规则 | 例如：回调失败后可重试；接口超时后订单保持原状态 |
| 排除范围 | 例如：本场景不覆盖支付；不覆盖后台人工改价 |
| 补充说明 | 例如：提交订单无二次确认弹窗；灰度开关需开启 |

#### 3.2 测试点

<!--
填写说明：
- 大类和子类按下方类型定义选择。
- 类型用于牵引测试分析覆盖面，不代表测试设计方法。
- 测试设计方法由 Test-Design-Agent 在后续设计阶段推导。
- 测试点应表达一个明确的规则、风险、路径或条件。
- 测试点应写到“要验证什么”的程度，不写成“怎么验证”。
- 测试点可以引用场景设计约束中的规则、角色、数据范围和观察点，但不展开具体执行路径。
- 测试点是后续用例设计的主覆盖目标；一个测试点可展开为 1-N 条测试用例，一条用例也可在追溯矩阵中覆盖同一场景下共享执行链路的多个测试点。
- 如果测试点已经包含完整操作步骤、测试数据和预期结果，说明粒度已接近测试用例，应上收为验证目标。
- 对需要使用边界值、等价类、判定表、组合、状态转换等方法展开的内容，只描述待验证规则或条件，不在测试点中展开方法产物。
- 不填写测试设计方法，例如边界值、判定表、状态转换。
- 不展开具体边界值、组合矩阵或状态迁移矩阵。
- 不填写测试步骤或完整预期结果。
- 如果一个测试点同时包含多个独立规则，应拆成多个测试点。

测试点类型定义：
| 大类 | 子类 | 定义 |
|---|---|---|
| 功能性 | 功能正确性测试 | 测试系统提供的功能是否能够正确、完全满足用户需要。 |
| 功能性 | 功能交互测试 | 通过系统不同功能之间的交互操作，验证系统是否满足需求预期；确认新功能的引入对继承功能的影响符合预期，如增强、兼容等。 |
| 功能性 | 协议一致性测试 | 测试系统是否符合相关协议或标准。 |
| 性能 | 性能规格测试 | 针对系统是否达到规定的性能规格进行测试，如容量、响应时间等。 |
| 性能 | 资源效率测试 | 验证系统完成功能或提供性能时，与所使用资源之间的比率是否达成设计预期，并确认是否满足目标网络场景的商用要求。 |
| 兼容性 | 配套兼容性测试 | 指对厂家内部两个或两个以上功能实体能否兼容运行、为客户提供既定功能和性能能力的测试。包括版本配套兼容性、网络兼容性、软件版本前向/后向兼容性、硬件一致性等，应包含但不限于软件版本差异、器件差异、单板差异、整机配合一致性。 |
| 兼容性 | 互通测试 | 不同厂商、不同类型设备或软件之间的协同工作测试。 |
| 易用性 | 用户体验测试 | 用户体验是指人们对于正在使用或期望使用的产品、系统或服务的认知印象和回应，覆盖使用前、使用中和使用后的全部感受。体验测试是指通过用户群体特征识别、用户使用场景识别、关键体验要素分析，从用户角度针对产品开展的一系列测试活动；其重点不是验证产品实现是否符合初始设计，而是验证产品是否为用户带来价值、解决用户困惑。 |
| 易用性 | 全球化测试 | 全球化指产品适用于各国的语言与文化，并以恰当方式实现国际化和本地化的程度，包括但不限于英文资料、时区、文化习惯等。 |
| 可靠性 | 容错容灾测试 | 通过人为在系统中引入故障，考察系统对故障的容错能力。故障注入不仅需要考察产品对故障的检测、定位和隔离能力，也要考察产品从故障中恢复的能力，即测试系统从硬件或软件故障中恢复的能力。 |
| 可靠性 | 过载测试 | 通过使系统达到或超过其设计最大负荷，检验系统在非正常状态下的特性表现。这里的特性主要指系统所需完成的正常功能。 |
| 可靠性 | 耐力测试 | 系统在模拟一定负载条件下进行长时间不间断测试，通常大于 48 小时，用于考察系统长时间持续运行能力，并识别系统在该条件下执行性能和状态的变化。又称长时间测试、持久测试或长期稳定性测试。 |
| 可靠性 | 可用性测试 | 测试系统处于按要求执行状态的能力。 |
-->

#### 5.1 接口设计约束

##### 必填约束

| 约束项 | 内容 |
|---|---|
| 接口入口 | 例如：POST /api/example |
| 调用方 | 例如：Web 前端、第三方系统、内部服务 |
| 请求条件 | 例如：鉴权方式、请求头、必填入参、业务数据状态 |
| 契约规则 | 例如：请求方式、Content-Type、入参、出参、错误码、幂等规则 |
| 观察点 | 例如：HTTP 状态、业务码、响应字段、数据状态、日志/审计 |

##### 可选约束

| 约束项 | 内容 |
|---|---|
| 关键调用 | 例如：正常请求、缺失必填参数、非法参数、重复请求 |
| 状态约束 | 例如：调用成功后数据状态变化；失败时数据保持原状态 |
| 禁止行为 | 例如：非法请求不得产生业务数据；无权限调用不得返回敏感字段 |
| 外部依赖 | 例如：鉴权服务可用；下游服务可用；消息队列可用 |
| 异常处理规则 | 例如：超时、服务异常、空数据、字段缺失时的响应格式 |
| 排除范围 | 例如：本接口不覆盖页面展示；不覆盖压测 |
| 补充说明 | 例如：错误码规则需与统一接口规范保持一致 |
