# 测试用例标准

## 字段

单条测试用例明细只包含以下字段：

| 字段 | 要求 |
|---|---|
| 用例编号 | 必填，默认 `TC-001` 起连续编号 |
| 用例名称 | 必填，表达被测对象、场景和验证目标 |
| 用例等级 | 必填，只允许 `Level 0`、`Level 1`、`Level 2`、`Level 3`、`Level 4` |
| 前置步骤 | 必填，描述执行前必须满足的账号、数据、环境、配置和状态 |
| 测试步骤 | 必填，使用有序列表描述可执行操作 |
| 预期结果 | 必填，与测试步骤逐条一一对应，数量保持一致 |

测试点关联、覆盖状态和备注只在追溯矩阵中表达。

## 编号规则

- 默认从 `TC-001` 开始连续编号。
- 同一份报告中不得重复编号。
- 删除或合并用例后，应重新保证编号连续。
- 若项目 memory 定义编号规则，以 memory 为准。

## 等级规则

| 用例等级 | 定位 | 执行建议 | 失败影响 |
|---|---|---|---|
| Level 0 | 核心。最核心的功能用例 | 建议每个转测试的版本都验证 | 执行失败意味着版本核心功能受损，没有继续验证的价值 |
| Level 1 | 关键。涉及系统特性的关键功能、关键可靠性等内容的用例 | 建议每个迭代验证 | 执行失败即特性内某个关键功能点测试出现阻塞问题 |
| Level 2 | 重要。覆盖系统所有重要功能的用例 | 建议 TR 点或对外发布版本进行完整验证 | 执行失败表示系统重要功能未正常实现 |
| Level 3 | 一般。版本全量测试较为完整的测试用例 | 建议版本变更较大、需要全量验证覆盖时，根据版本修改点涉及范围，选择变更较大的特性相关部分或全部用例作为回归范围 | 执行失败表示相关一般功能或回归范围存在问题 |
| Level 4 | 生僻。对于较生僻的应用场景和预置条件，是版本不常用场景的用例 | 建议新特性首次验证使用；验证通过后，大多数情况不需要回归执行 | 执行失败表示低频场景或特殊预置条件存在问题 |

## 命名规则

推荐结构：

```text
验证<被测对象>在<场景/条件>下<预期目标>
```

示例：

- 验证普通用户在有效输入下成功提交订单
- 验证审批单在已撤回状态下不允许重复提交
- 验证账期跨月时系统按新账期生成结算记录

## 粒度规则

- 一条用例只验证一个清晰目标。
- 同一业务链路中的多个必要动作可以放在一条用例中。
- 相互独立的规则、异常和状态迁移应拆成不同用例。
- 高风险测试点可派生多条用例，低风险测试点应避免过度扩展。
