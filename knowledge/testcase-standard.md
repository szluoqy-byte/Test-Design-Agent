# 测试用例标准

## 字段

单条测试用例明细只包含以下字段：

| 字段 | 要求 |
|---|---|
| 用例编号 | 必填，默认 `TC-001` 起连续编号 |
| 用例名称 | 必填，表达被测对象、场景和验证目标 |
| 用例等级 | 必填，只允许 `Level 0`、`Level 1`、`Level 2`、`Level 3`、`Level 4` |
| 前置步骤 | 必填，描述执行前必须满足的账号、数据、环境、配置和状态 |
| 测试步骤 | 必填，使用有序列表描述可执行操作 |
| 预期结果 | 必填，与测试步骤逐条或分组对应 |

测试点关联、覆盖状态和备注只在追溯矩阵中表达。

## 编号规则

- 默认从 `TC-001` 开始连续编号。
- 同一份报告中不得重复编号。
- 删除或合并用例后，应重新保证编号连续。
- 若项目 memory 定义编号规则，以 memory 为准。

## 等级规则

| 测试点级别或风险 | 默认用例等级 | 说明 |
|---|---|---|
| Level 0 / 极高风险 / 资金安全 / 数据破坏 | Level 0 | 阻断发布或必须优先执行 |
| Level 1 / 高风险 / 核心主链路 | Level 1 | 核心回归和主流程优先执行 |
| Level 2 / 中风险 / 常规功能 | Level 2 | 常规测试范围 |
| Level 3 / 低风险 / 辅助功能 | Level 3 | 补充测试或抽测 |
| Level 4 / 很低风险 / 展示性检查 | Level 4 | 低频执行或候选补充 |

## 命名规则

推荐结构：

```text
验证<被测对象>在<场景/条件>下<预期目标>
```

示例：

- 验证普通用户在有效输入下成功提交订单
- 验证审批单在已撤回状态下不允许重复提交
- 验证账期跨月时系统按新账期生成结算记录

## 粒度规则

- 一条用例只验证一个清晰目标。
- 同一业务链路中的多个必要动作可以放在一条用例中。
- 相互独立的规则、异常和状态迁移应拆成不同用例。
- 高风险测试点可派生多条用例，低风险测试点应避免过度扩展。
