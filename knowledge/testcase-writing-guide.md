# 可执行测试用例编写准则

## 前置步骤

前置步骤用于准备执行条件，应包含：

- 账号、角色或权限状态。
- 业务数据状态。
- 环境、渠道、版本或配置开关。
- 依赖系统或定时任务状态。

编写前置步骤时，应优先使用输入中“前置条件”“执行用户/角色”“测试数据因子”“业务设计约束”和“补充说明”提供的信息；前置条件提供环境、配置、依赖服务、账号权限和数据准备方式，测试数据因子中的状态集合、枚举值、取值区间和状态流转用于指导用例设计展开，但单条用例中仍需落成明确的前置条件或操作条件。

避免写成泛化描述，例如“准备测试环境”“准备测试数据”。如果缺少项目细节，可写“准备一个满足条件的有效用户”，并在待确认问题中标注缺口。

## 测试步骤

测试步骤必须：

- 使用有序列表。
- 每一步有明确操作对象。
- 按真实执行顺序排列。
- 不把预期结果混入操作步骤。
- 不使用“验证功能正常”“检查结果正确”等空泛描述。
- 页面入口类步骤必须尽量写成可点击路径，例如“点击【一级菜单】 > 【二级页面】 > 【目标页签】”。
- 如果输入的场景测试条件提供了场景入口/触发方式，测试步骤应优先使用该入口中的菜单、页面、页签、按钮名称，不要退化成“进入某菜单”“进入某入口”这类泛化描述。
- 每条 GUI 场景测试用例都应在测试步骤中完整引用入口操作链路，例如“点击【一级入口】 > 【二级入口】 > 【三级入口】 > 【目标页面】进入目标页面”。不要只在前置步骤中写“已进入目标页面”后省略入口操作。
- 如果 context-pack 基于 domain memory 提供了补全后的建议入口路径，而当前输入只给出较短入口、目标页面或页签，应优先使用补全后的完整路径。只有当前输入在同一层级明确给出不同入口时，才按冲突处理并进入待确认问题。

常见交互步骤应写完整：

- 选择类控件：打开控件、选择目标选项、必要时清除选择、收起控件，并观察选中态或结果变化。
- 查询筛选类操作：输入或选择筛选条件、触发查询、观察结果与筛选条件是否一致、观察筛选条件是否保持。
- 分页类操作：触发查询或加载列表、观察分页组件状态、切换页码或页容量、观察列表结果和筛选条件是否保持。
- 故障或异常类操作：如果故障注入、Mock、断网、超时配置或服务开关是测试人员需要执行的动作，应写入测试步骤；前置步骤只描述执行前已具备的环境或数据条件。

### GUI 场景测试步骤

GUI 场景测试适用于页面、菜单、页签、表单、输入框、下拉框、按钮、弹窗、表格、分页等用户界面交互；接口测试不套用 GUI 控件写法，应按接口契约、字段、状态码、错误码、鉴权、幂等或性能目标编写。

GUI 场景测试步骤应使用“操作对象 + 用户动作 + 必要输入”的公共语言，不写成检查意图。避免只写“查看页面”“验证布局”“检查列表”“执行查询”。如果一个页面检查包含多个观察对象，应拆成多个步骤，例如分别查看筛选区、结果区、分页区或提示区，保持步骤与预期结果逐条对应。

GUI 场景测试用例应从可执行入口开始写：先使用具备权限的用户登录或到达约定起始页面，再点击完整菜单、页面或页签路径进入目标功能区。即使某条用例的验证重点是列表、筛选、分页、弹窗或状态展示，也应保留入口链路，便于人工执行和后续 GUI 自动化脚本复用。

常见 GUI 动作应按下列方式表达：

| 对象 | 步骤关注 | 预期关注 |
|---|---|---|
| 页面入口/页签 | 打开目标页面或点击目标页签 | 目标页面或页签展示，当前页签选中，目标功能区域可见 |
| 输入框 | 清空输入框，输入指定值，必要时触发查询或提交 | 输入值展示、长度或格式限制生效、触发后的结果与输入条件一致 |
| 下拉/单选控件 | 打开控件，查看选项，选择目标选项 | 选项完整、顺序正确、文案完整、选中态正确 |
| 多选控件 | 打开控件，选择多个目标选项，必要时清除选择 | 多个选项可同时选中，选中值展示或保持，清除后恢复未选择状态 |
| 按钮 | 点击目标按钮 | 按钮状态、触发结果、加载状态或错误提示符合设计 |
| 表格/列表 | 查看表头、行数据、空态或刷新后的结果 | 字段完整、字段顺序符合设计、结果范围符合条件、空态可理解 |
| 分页区域 | 查看总数和页码，切换页码或页容量 | 总数、页码、页容量和列表数据同步变化，筛选条件保持 |
| 弹窗/提示 | 触发弹窗或提示，确认、取消或关闭 | 弹窗标题、正文、按钮、关闭后状态和提示文案可观察可判定 |

GUI 场景测试预期应尽量写成可断言的界面状态，例如元素存在、文案展示、按钮可用或禁用、选项顺序、选中态、表格字段、表格行范围、空态提示、分页总数、筛选条件保持、错误提示。不要只写“页面正常”“结果正确”“符合设计”。

可接受的中性步骤：

- 使用具备目标功能权限的用户登录系统。
- 点击【一级入口】 > 【二级入口】 > 【三级入口】 > 【目标页面】进入目标页面。
- 点击【目标页签】进入目标页签。
- 选择一条满足测试条件的业务数据并触发目标操作。

如果具体菜单路径、按钮名称或弹窗文案缺失，不编造；可以写“通过已确认的目标功能入口打开目标页面”，并在待确认问题中记录缺失的菜单路径或按钮名称。

## 预期结果

预期结果必须可观察、可判定。可观察对象包括：

- 页面展示、按钮状态、弹窗提示。
- 接口响应、错误码、返回字段。
- 数据库或后台记录状态。
- 消息、通知、日志、审计记录。
- 权限拦截、数据隔离、状态保持。

预期结果必须与测试步骤逐条一一对应，数量保持一致。不要把多个步骤合并成一条笼统预期，也不要只写“成功”“失败”“结果正确”。

推荐写法：

```text
测试步骤：
1. 使用具备目标功能权限的用户登录系统。
2. 点击【一级入口】 > 【二级入口】 > 【三级入口】 > 【目标页面】进入目标页面。
3. 点击【目标页签】进入目标页签。

预期结果：
1. 系统展示已登录状态。
2. 系统展示目标页面，页面未出现加载失败提示。
3. 系统切换到目标页签，并展示该页签对应的功能区域。
```

## 测试数据

测试数据不作为单条用例独立字段输出，但必须在前置步骤或测试步骤中显式体现关键条件，例如：

- 边界值：金额等于 0、等于上限、超过上限。
- 角色：普通用户、管理员、无权限用户。
- 状态：待审批、已撤回、已取消、已超时。
- 周期：账期开始、账期中、账期结束、跨账期。
- 列表数量：小于分页容量、等于分页容量、大于分页容量。
- 组合条件：全部条件满足、仅部分条件满足、全部条件不满足。
- 异常返回：空数据、字段缺失、格式异常、超时或服务失败。

同一控件或同一路径需要覆盖多个输入值时，测试步骤中应显式列出输入值清单或选择项清单。若不同输入值的预期差异明显、风险不同或失败定位方向不同，应拆成多条用例；若路径相同、风险相近且预期可按清单逐项表达，可以在一条用例中形成参数化步骤。

参数化步骤应避免只写“分别输入多个值”。推荐写法：

```text
测试步骤：
1. 在目标输入框中依次使用输入值清单执行查询：已存在的有效值 A、不存在的有效值 B、包含特殊字符的值 C、达到长度上限的值 D、超过长度上限的值 E。

预期结果：
1. 输入 A 时结果区域展示匹配记录；输入 B 时结果区域展示空态；输入 C 时系统按设计完成查询或提示格式限制；输入 D 时可完成输入和查询；输入 E 时超出部分被限制输入或展示设计要求的提示。
```

## 待确认信息

以下信息缺失时应进入待确认问题：

- 必须知道的菜单路径、入口、按钮或弹窗动作。
- 必须知道的账号、角色、权限边界。
- 必须知道的业务规则、状态机或异常处理规则。
- 高风险用例的预期结果无法可靠判定。
