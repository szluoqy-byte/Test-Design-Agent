# 测试场景、测试点与测试用例边界标准

本文档用于定义 Test-Design-Agent 输入和设计过程中的核心层级边界，帮助判断需求、测试场景、场景测试条件、测试点和测试用例之间的关系。该标准用于输入规范化、输入评审、测试点粒度检查和测试用例生成，不替代具体测试设计模式。

## 分层关系

推荐结构如下：

```text
需求
  -> 测试场景
      -> 场景测试条件
      -> 测试点
          -> 测试用例
```

生成顺序应从需求出发：

1. 从需求中识别业务场景。
2. 为每个测试场景补充场景测试条件。
3. 在测试场景下拆解测试点。
4. 以测试点作为主覆盖目标，结合设计模式生成测试用例，并在追溯矩阵中表达测试点与用例的覆盖关系。

不要先罗列测试点，再反向聚类成测试场景。测试场景是从需求中的业务流程、用户任务、系统触发或外部交互中识别出来的验证范围。

## 核心定义

| 对象 | 定义 | 解决的问题 |
|---|---|---|
| 需求 | 产品、系统或业务规则的来源说明，包含功能、流程、约束、质量要求或验收标准 | 要验证什么业务变化或规则 |
| 测试场景 | 从需求中识别出的一个完整业务使用或系统触发情境，描述执行主体在特定入口、流程或触发条件下完成某个业务目标 | 这段需求在什么业务情境下需要被验证 |
| 场景测试条件 | 绑定在测试场景上的共用执行条件和业务设计约束 | 后续生成可执行用例需要哪些共用信息 |
| 测试点 | 在测试场景下进一步拆解出的验证目标，描述该场景中需要验证的规则、条件、路径、风险或质量关注点 | 这个场景中有哪些内容必须被验证 |
| 测试用例 | 由测试点结合测试设计模式生成的可执行验证步骤，包含前置步骤、测试步骤和预期结果 | 具体怎么测、准备什么、执行什么、观察什么 |

## 测试场景边界

测试场景应描述“谁/什么系统动作，在什么入口、流程或触发条件下，完成什么业务目标”。

合适的测试场景通常满足以下特征：

- 来源于需求中的业务流程、用户任务、系统触发、外部交互或端到端业务链路。
- 具有明确执行主体，例如用户、管理员、定时任务、外部系统或系统组件。
- 具有明确入口、流程或触发条件。
- 具有明确业务目标。
- 场景下的测试点能够共用主要入口、角色、前置条件、测试数据因子和业务设计约束。

如果执行主体、入口/触发方式、业务目标或核心规则明显不同，应拆成不同测试场景。

### 测试场景不是

| 写法 | 更可能属于 | 原因 |
|---|---|---|
| 库存不足时阻止提交订单 | 测试点 | 它描述一个规则或条件，不是完整业务情境 |
| 收货地址为空提示错误 | 测试点或用例目标 | 它描述一个校验目标，缺少业务流程范围 |
| 点击提交订单按钮 | 测试步骤 | 它描述操作动作，不是验证范围 |
| 库存等于 0 | 测试数据 | 它描述具体数据取值，不是场景 |
| 普通用户从商品详情页提交订单 | 测试场景 | 它包含执行主体、入口/流程和业务目标 |

## 场景测试条件边界

场景测试条件绑定在测试场景上，用于承载该场景下测试点共用的信息。

场景测试条件适合包含：

- 场景入口/触发方式：页面路径、菜单入口、接口入口、任务触发方式。
- 执行用户/角色：具体用户、账号、管理员、系统任务、外部系统、权限身份。
- 前置条件：执行该场景测试前必须具备的共用条件，例如环境、配置开关、依赖服务、测试账号权限或数据准备方式。
- 测试数据因子：场景下已知会影响规则判断、路径分支、权限结果、状态流转或观察结果的数据对象、字段、状态集合或取值区间；它承载需求已明确的数据约束，不展开边界值、等价类、组合矩阵或具体用例数据。
- 业务设计约束：需求或设计文档中已明确的业务规则、允许/禁止行为、状态流转、权限限制、数量范围或处理原则。

需求已明确的状态集合、枚举值、取值区间和状态流转应尽量写入测试数据因子；已部署版本、服务可用、前端资源可加载、测试账号具备权限、数据可预置等执行前提应写入前置条件；无法确认的状态、范围、账号、配置、数据准备方式或接口契约信息，应写入待确认信息。补充说明只承载不适合固定成表格的背景，例如可观察结果位置或特殊假设，不用于承载本次不覆盖内容。

场景测试条件中的前置条件是场景级共用前提，不应写成单条测试用例的完整前置步骤；测试数据因子不应展开具体边界值、组合矩阵或状态迁移矩阵。它提供设计输入，不替代测试用例。

## 测试点边界

测试点应写到“要验证什么”的程度，不写成“怎么验证”。

合适的测试点通常满足以下特征：

- 挂在某个测试场景下。
- 表达一个明确的规则、风险、路径、条件或质量关注点。
- 可以引用场景测试条件中的业务设计约束、前置条件、角色和测试数据因子。
- 不展开具体执行路径、测试步骤、测试数据取值或完整预期结果。
- 通常作为后续设计阶段生成测试用例的主覆盖目标；一个测试点可展开成 1-N 条测试用例，一条测试用例也可以在追溯矩阵中覆盖多个测试点。

测试点不应包含：

- 测试设计方法名称，例如边界值、判定表、状态转换、组合测试。
- 已展开的方法产物，例如具体边界值集合、组合矩阵、状态迁移矩阵。
- 具体操作步骤，例如“点击 A，再点击 B”。
- 完整预期结果，例如“页面提示库存不足且数据库不生成订单记录”。

如果一个测试点同时包含多个独立规则，应拆成多个测试点。

## 测试用例边界

测试用例是可执行产物，必须包含：

- 用例编号。
- 用例名称。
- 用例等级。
- 前置步骤。
- 测试步骤。
- 预期结果。

当一条内容已经包含具体前置、操作步骤、测试数据和可判定预期时，它已经接近测试用例，不应再作为测试点输入。

## 测试点与测试用例关系

测试点到测试用例不是严格的一对一关系。更准确的关系是：测试点是用例设计的主覆盖目标，测试用例是可执行验证产物，二者通过追溯矩阵表达覆盖关系。

默认设计策略：

- 以一个测试点作为主覆盖目标生成测试用例。
- 一个测试点可以根据风险、边界、异常、组合、状态或周期规则展开为 1-N 条测试用例。
- 一条测试用例可以覆盖多个测试点，但这些测试点必须属于同一测试场景，且共享同一执行链路、前置条件和主要观察点。
- 多个测试点被一条用例覆盖时，应在追溯矩阵中表达，不写入单条测试用例明细。

允许一条测试用例覆盖多个测试点的情况：

- 多个测试点是同一业务事务中的不同观察面，例如页面结果、数据记录、状态变化或通知结果。
- 多个测试点共享同一前置、同一操作链路，并且一次执行可以自然观察到多个结果。
- 被合并覆盖的测试点不是相互独立的分支、异常、规则或风险。

不应合并为一条测试用例的情况：

- 测试点对应不同条件、不同分支、不同异常或不同状态迁移。
- 测试点失败后的缺陷定位方向不同。
- 测试点等级差异明显，合并后会掩盖执行优先级。
- 为了减少用例数量，把多个独立验证目标串成一条长用例。

## 粒度判断

| 判断问题 | 倾向层级 |
|---|---|
| 这段内容是否来自需求中的一个业务使用、系统触发或外部交互情境？ | 测试场景 |
| 是否能说明执行主体、入口/触发方式和业务目标？ | 测试场景 |
| 是否能承载一组共用入口、角色、前置条件、测试数据因子和业务设计约束？ | 测试场景 |
| 是否只表达一条需要验证的规则、风险、路径或条件？ | 测试点 |
| 是否已经写出具体数据、操作步骤和预期结果？ | 测试用例 |
| 是否只是一个按钮、菜单、字段或取值？ | 通常不是场景，需归入约束、步骤或数据 |

## 示例

### 需求

```text
普通用户可以从商品详情页提交实物商品订单。库存充足、商品上架且收货地址有效时允许提交订单；库存不足、商品下架或收货地址缺失时应阻止提交。
```

### 测试场景

| 场景 ID | 场景名称 | 场景目标 |
|---|---|---|
| SC-001 | 普通用户提交实物商品订单 | 验证普通用户在商品详情页提交实物商品订单时，系统按库存、地址和商品状态规则完成提交或拦截 |

### 场景测试条件

| 条件项 | 内容 |
|---|---|
| 场景入口/触发方式 | 首页 > 商品详情页 > 立即购买 > 订单确认页 |
| 执行用户/角色 | 普通用户 |
| 前置条件 | 用户已登录并具备下单权限；库存服务可用；测试商品、库存和收货地址可按用例准备 |
| 测试数据因子 | 商品状态：上架/下架；库存数量：0-9999；收货地址状态：有效/缺失；用户下单权限：有/无；订单状态：待支付 |
| 业务设计约束 | 库存充足、商品上架且收货地址有效时允许提交；否则阻止提交；提交成功后订单进入待支付 |

### 测试点

| 测试点 ID | 测试点 |
|---|---|
| TP-001 | 库存充足、商品上架且收货地址有效时允许提交订单 |
| TP-002 | 库存不足时阻止提交订单 |
| TP-003 | 收货地址缺失时阻止提交订单 |
| TP-004 | 商品下架后阻止提交订单 |
| TP-005 | 重复提交时不得生成重复订单 |

### 测试用例

| 用例编号 | 用例名称 | 用例等级 |
|---|---|---|
| TC-001 | 验证普通用户在库存充足且地址有效时成功提交实物商品订单 | Level 0 |
| TC-002 | 验证普通用户在库存不足时提交实物商品订单被拦截 | Level 1 |

测试用例还必须继续补充前置步骤、测试步骤和预期结果，不能只停留在名称和等级。

## 常见误区

| 误区 | 问题 | 调整方式 |
|---|---|---|
| 把每条规则都写成测试场景 | 场景过细，无法承载共用约束 | 将规则放入同一业务场景下的测试点 |
| 先写一堆测试点再归类场景 | 场景来源被反转，容易脱离需求业务流程 | 先从需求识别场景，再拆测试点 |
| 测试点写成具体操作步骤 | 粒度过细，提前进入用例编写 | 上收为验证目标 |
| 测试点写成测试方法 | 混淆测试点分析和测试设计阶段 | 测试点只写待验证规则，设计方法由 Test-Design-Agent 选择 |
| 场景测试条件缺少业务设计约束 | 后续用例难以写出可判定预期 | 在场景测试条件中补充需求或设计文档已明确的业务规则、状态流转、权限限制或处理原则 |

## 使用规则

- 输入规范化时，用本文档检查测试场景、测试点和测试用例是否混写。
- 测试用例设计时，测试点如果过粗，应在待确认问题或设计方案中说明拆分建议。
- 测试点如果已经包含步骤和预期，应上收为验证目标，再由 `testcase-writing` 重新生成可执行用例。
- 模板可以保留轻量引导，详细边界以本文档为准。
