# 基于风险的测试设计

## 方法定义

基于风险的测试设计根据失败概率、业务影响、可恢复性、用户影响范围和历史缺陷情况，决定测试深度、用例等级、覆盖优先级和补充场景。它是上层策略，通常叠加在基于规格的主模式之上。

风险设计不直接替代判定表、状态转换或场景测试；它决定哪些规则必须更深、更早、更严格地测。

## 适用场景

- 资金、安全、合规、隐私、权限、审计。
- 不可逆操作，例如删除、结算、支付、发放、关闭。
- 高用户影响或核心主链路。
- 历史高发缺陷或线上事故区域。
- 失败后难恢复、难发现或影响扩散快的功能。

## 风险评估维度

| 维度 | 判断问题 |
|---|---|
| 业务影响 | 失败是否影响资金、客户、合规或核心流程 |
| 发生概率 | 该场景是否复杂、变更多、历史缺陷多 |
| 可检测性 | 失败是否容易被发现 |
| 可恢复性 | 失败后是否能回滚、补偿或修复 |
| 影响范围 | 影响单用户、批量用户还是全局系统 |
| 安全合规 | 是否涉及越权、敏感数据、审计责任 |

## 设计步骤

1. 识别测试点的风险对象和失败后果。
2. 判断风险等级并映射用例等级。
3. 选择主规格模式生成基础用例。
4. 对高风险点补充以下场景：
   - 主路径成功。
   - 关键异常拦截。
   - 边界和临界条件。
   - 权限和防护。
   - 幂等、补偿、审计。
5. 调整用例执行优先级和覆盖深度。
6. 对无法可靠验证的高风险信息生成待确认问题。

## 用例派生规则

| 风险类型 | 补充用例方向 |
|---|---|
| 资金风险 | 金额边界、重复扣款、退款补偿、审计记录 |
| 权限风险 | 越权访问、角色切换、租户隔离、数据归属 |
| 合规风险 | 必填留痕、审批链路、敏感数据展示 |
| 不可逆操作 | 二次确认、撤销限制、误操作防护 |
| 高用户影响 | 降级、回滚、批量失败、通知 |
| 历史高发 | 回归用例、异常保护、日志追踪 |

## 等级映射

| 风险判断 | 建议等级 |
|---|---|
| 失败会导致版本核心功能受损，且没有继续验证价值 | Level 0 |
| 失败会阻塞特性内关键功能、关键可靠性或关键业务链路 | Level 1 |
| 失败会影响系统重要功能，需要在 TR 点或对外发布版本完整验证 | Level 2 |
| 失败影响一般功能，适合在版本变更较大或全量验证时纳入回归 | Level 3 |
| 失败只影响低频应用场景或特殊预置条件，适合新特性首次验证后按需回归 | Level 4 |

## 数量控制

- 高风险测试点可以从一个测试点派生多条用例。
- 低风险测试点不因为风险模式叠加而扩大用例数量。
- 风险补充用例必须有独立失败后果，不能只是换个名称。
- 如果风险只能通过监控或审计验证，应在预期中明确观察对象。

## 前置步骤关注

- 准备高风险对象，例如金额、权限、敏感数据、批量数据。
- 明确操作是否可逆，是否需要二次确认。
- 准备审计、日志或补偿记录的观察入口。

## 测试步骤关注

- 覆盖真实风险触发动作。
- 对不可逆操作，明确确认或取消动作。
- 对权限风险，使用不同角色或租户触发。
- 对资金风险，关注重复、边界和失败恢复。

## 预期结果关注

- 风险被阻断或被正确控制。
- 数据不被破坏、不重复、不泄露。
- 审计、日志、通知和补偿记录完整。
- 失败时系统保持安全状态。
- 成功时关键业务结果准确。

## 常见误区

- 把“高风险”只体现为高等级，不增加覆盖深度。
- 只测成功路径，忽略防护、审计和补偿。
- 生成大量低价值风险用例，挤占核心场景。
- 未说明风险失败后果，导致等级缺乏依据。

## 设计方案片段

| 风险信号 | 用例标题建议 | 预期关注 |
|---|---|---|
| 资金 | 验证支付请求重复提交时不会重复扣款 | 扣款幂等且有审计记录 |
| 权限 | 验证无权限用户访问他人订单时被拦截 | 数据不泄露且记录访问失败 |
| 不可逆 | 验证删除关键数据前系统要求二次确认 | 取消确认时数据保持不变 |
