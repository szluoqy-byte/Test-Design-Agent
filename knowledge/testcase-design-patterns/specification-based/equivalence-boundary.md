# 等价类与边界值测试设计

## 方法定义

等价类划分将输入或业务条件划分为若干“系统应以相同方式处理”的集合，从每个集合中选择代表值生成用例。边界值分析关注集合边界附近最容易出错的位置，例如最小值、最大值、刚好等于、刚好小于、刚好大于。

在用例设计中，等价类和边界值通常一起使用：先划分有效/无效等价类，再为关键等价类选择边界值。

## 适用场景

- 数值范围：金额、数量、年龄、比例、库存、次数。
- 长度范围：文本长度、文件大小、列表数量。
- 时间范围：开始时间、结束时间、有效期、截止时间。
- 枚举集合：状态、类型、渠道、角色、币种。
- 格式规则：手机号、邮箱、证件号、编码、URL。
- 空值和缺省值：必填、选填、默认值。

## 不适用场景

- 多个条件组合共同决定结果，应优先使用判定表。
- 对象生命周期或状态迁移，应优先使用状态转换。
- 周期性批处理或账期问题，应优先使用处理周期。
- 端到端业务路径，应优先使用场景测试。

## 建模输入

从测试点中抽取：

| 元素 | 说明 |
|---|---|
| 参数/条件 | 被划分的输入项或业务条件 |
| 有效范围 | 系统允许并正常处理的取值 |
| 无效范围 | 系统应拒绝、提示或忽略的取值 |
| 边界点 | 最小、最大、临界、刚好等于、刚好越界 |
| 处理结果 | 每类输入对应的预期行为 |

## 设计步骤

1. 明确被测对象和输入域。
2. 列出有效等价类，例如合法金额、合法日期、支持的枚举值。
3. 列出无效等价类，例如空值、非法格式、越界值、不支持枚举。
4. 识别边界点：`min`、`min - 1`、`min + 1`、`max - 1`、`max`、`max + 1`。
5. 对每个高风险边界生成独立用例。
6. 对低风险等价类选择代表值，不做无意义穷举。
7. 若多个参数同时存在，只保留一个参数变化，其他参数保持有效默认值；多参数组合交给 DCoT。

## 用例派生规则

| 类型 | 派生方式 |
|---|---|
| 有效等价类 | 每类至少 1 条代表性成功用例 |
| 无效等价类 | 每类至少 1 条拦截、提示或拒绝用例 |
| 下边界 | 覆盖最小值、低于最小值 |
| 上边界 | 覆盖最大值、高于最大值 |
| 缺省/空值 | 覆盖必填为空、选填为空、默认值 |
| 格式 | 覆盖合法格式、非法格式、特殊字符 |
| 容量边界 | 覆盖小于容量、等于容量、大于容量 |

## 数量控制

- `Level 0`：覆盖有效代表值、关键上下边界、关键无效类。
- `Level 1`：覆盖有效代表值和主要无效边界。
- `Level 2`：覆盖一个有效类和一个最关键无效类。
- `Level 3/4`：只覆盖最有代表性的有效或无效路径。

## 前置步骤关注

- 准备满足其他条件的有效业务对象。
- 固定与当前参数无关的配置、角色和状态。
- 明确边界数据来源，避免步骤中临时造数不可执行。

## 测试步骤关注

- 明确输入哪个字段或选择哪个枚举值。
- 每条用例只改变一个核心参数，便于定位失败原因。
- 对非法输入说明提交、保存或触发校验的动作。

## 预期结果关注

- 合法输入被接受并产生正确业务结果。
- 非法输入被拒绝，提示内容明确。
- 越界数据不被保存，不产生副作用。
- 边界值按规则归属到正确区间。

## 常见误区

- 只测最大值和最小值，不测刚好越界。
- 多个参数同时取边界，导致失败原因不可定位。
- 把每个枚举值都展开成用例，造成低价值膨胀。
- 缺少对空值、默认值和非法格式的考虑。
- 列表、分页、批量处理等容量类规则只测“大于容量”，遗漏“小于容量”和“等于容量”的边界。

## 设计方案片段

| 覆盖意图 | 用例标题建议 | 预期关注 |
|---|---|---|
| 有效等价类 | 验证用户输入合法金额时提交成功 | 成功保存，金额展示正确 |
| 下边界 | 验证金额等于最小值时提交成功 | 边界值被接受 |
| 越界值 | 验证金额低于最小值时提交失败 | 提示越界且不保存 |
