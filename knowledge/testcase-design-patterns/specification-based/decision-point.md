# 判定点测试设计 DPT

## 方法定义

判定点测试设计用于处理流程或规则中的单个判断节点。它关注某个条件是否成立，以及系统在通过、拒绝、默认或异常分支上的行为。

DPT 比判定表更轻量，适合条件较少、动作结果明确、不需要展开完整条件矩阵的场景。

## 适用场景

- 单个准入条件、校验点、开关点或拦截点。
- 判断某个状态是否允许操作。
- 判断某个字段是否满足规则。
- 判断配置开关开启或关闭后的处理。
- 流程中存在通过/拒绝、是/否、有效/无效分支。

## 不适用场景

- 多个条件组合共同决定多个动作，应使用 DTT。
- 判断点背后是对象状态机，应使用 STT。
- 判断点发生在周期任务中并与周期边界相关，应使用 PCT。

## 建模输入

| 元素 | 说明 |
|---|---|
| 判定点 | 发生判断的位置，例如提交、保存、审批、回调 |
| 判定条件 | 判断依据，例如库存是否充足、是否有权限 |
| 分支 | 通过、拒绝、默认、异常 |
| 触发动作 | 用户或系统触发判定的操作 |
| 可观察结果 | 页面、接口、状态、日志、消息 |

## 设计步骤

1. 明确判定点发生在哪一步。
2. 明确判定条件和判定结果。
3. 设计通过分支用例。
4. 设计拒绝或失败分支用例。
5. 识别默认分支、空值分支或异常分支。
6. 如果判定条件有边界值，叠加等价类与边界值。
7. 如果判定点风险高，补充审计、状态保持和副作用验证。

## 用例派生规则

| 分支类型 | 派生方式 |
|---|---|
| 通过分支 | 条件满足，业务继续流转 |
| 拒绝分支 | 条件不满足，系统阻止操作 |
| 默认分支 | 未配置或未命中特定条件时按默认规则处理 |
| 异常分支 | 条件数据缺失、依赖失败或非法输入 |
| 边界分支 | 条件刚好满足或刚好不满足 |

## 数量控制

- 普通判定点至少覆盖通过和拒绝。
- 高风险判定点补充边界、默认和异常。
- 多个相同规则的判定点可选一个代表场景。
- 如果分支过多，应判断是否升级为判定表。

## 前置步骤关注

- 准备满足或不满足判定条件的数据。
- 明确判定所需角色、状态、配置或库存。
- 对依赖外部系统的判定，准备依赖成功和失败状态。

## 测试步骤关注

- 操作应准确触发该判定点。
- 每条用例只覆盖一个核心分支。
- 拒绝分支应继续观察是否阻止后续操作。

## 预期结果关注

- 分支选择正确。
- 提示文案明确。
- 被拒绝时业务状态不被错误改变。
- 通过时进入正确后续流程。
- 异常分支有日志、告警或错误码。

## 常见误区

- 只测通过分支，不测拒绝分支。
- 把多个独立判定点混成一条用例。
- 拒绝后只看提示，不看状态是否被错误修改。
- 判定条件有多个输入时仍使用 DPT，遗漏组合关系。

## 设计方案片段

| 覆盖意图 | 用例标题建议 | 预期关注 |
|---|---|---|
| 通过分支 | 验证库存充足时用户可以提交订单 | 订单成功生成 |
| 拒绝分支 | 验证库存不足时用户提交订单被拦截 | 提示库存不足且不生成订单 |
| 默认分支 | 验证未配置特殊规则时按默认规则处理 | 命中默认处理结果 |
