# 数据组合测试设计 DCoT

## 方法定义

数据组合测试设计用于处理多个输入因子共同影响结果的场景。它通过识别因子、取值和约束，选择合适的组合策略生成代表性用例。组合策略可以是全组合、pairwise、正交组合或基于风险的裁剪组合。

DCoT 的目标不是穷举所有可能，而是在成本可控的前提下覆盖关键因子交互。

## 适用场景

- 多个输入参数共同影响处理结果。
- 多配置项、多渠道、多版本、多角色组合。
- 平台、浏览器、设备、语言、地区组合。
- 业务条件之间存在少量约束，但不形成复杂动作矩阵。
- 测试点明确要求组合、兼容、适配或覆盖配置差异。

## 不适用场景

- 多条件直接映射到多个动作，应优先使用判定表。
- 只有一个判断节点，应优先使用判定点。
- 状态迁移或周期处理不应强行建成组合表。

## 建模输入

| 元素 | 说明 |
|---|---|
| 因子 | 影响结果的参数、配置、角色、渠道或状态 |
| 水平 | 每个因子的可选取值 |
| 约束 | 不允许组合、依赖组合、互斥组合 |
| 风险组合 | 高风险、历史缺陷、核心客户或主链路组合 |
| 预期差异 | 不同组合下结果是否相同或不同 |

## 设计步骤

1. 列出所有因子，剔除不会影响结果的装饰性维度。
2. 为每个因子选择有限水平，避免把无限输入域直接放入组合。
3. 标记非法组合和业务不可能组合。
4. 选择组合策略：
   - 因子少且风险高：全组合。
   - 因子多且交互风险一般：pairwise。
   - 因子水平较规整：正交组合。
   - 存在明确高风险组合：风险裁剪组合。
5. 强制加入历史缺陷组合、核心链路组合和边界组合。
6. 为每个组合生成可执行用例或把多个低风险组合合并为检查清单。

## 用例派生规则

| 组合类型 | 派生方式 |
|---|---|
| 基准组合 | 一条主路径成功用例 |
| 高风险组合 | 独立用例，等级不低于对应测试点 |
| 历史缺陷组合 | 独立回归用例 |
| 非法组合 | 拦截或不允许配置用例 |
| 兼容组合 | 结果一致性或差异性验证用例 |
| 多条件筛选组合 | 覆盖全部条件满足、仅部分条件满足、全部条件不满足，验证结果只包含命中组合的数据 |

## 数量控制

- 优先覆盖因子之间的二元交互。
- 不对低风险展示类组合做全组合。
- 同结果、同路径、同风险的组合可合并。
- `Level 0/1` 保留高风险组合和主链路组合。
- `Level 2/3/4` 优先使用 pairwise 或代表性组合。

## 前置步骤关注

- 明确组合所需的账号、角色、配置、版本或渠道。
- 对不可同时满足的组合提前标记为非法组合。
- 准备基准数据，避免组合差异被无关数据干扰。

## 测试步骤关注

- 明确设置或选择每个因子的取值。
- 组合验证动作应保持一致，便于比较结果。
- 若组合发生在不同平台或渠道，应说明切换方式。

## 预期结果关注

- 同类组合结果一致。
- 差异组合符合规格说明。
- 非法组合被拦截或不可配置。
- 高风险组合不产生数据错乱、权限穿透或状态异常。

## 常见误区

- 把无限数据值当成组合水平，导致无法收敛。
- 只生成 pairwise，漏掉业务指定的高风险三元组合。
- 不记录非法组合原因，导致用例不可执行。
- 将组合测试与判定表混用，条件和动作关系不清。
- 组合筛选只验证全部条件同时满足，遗漏部分条件满足或全部不满足时的排除效果。

## 设计方案片段

| 覆盖意图 | 用例标题建议 | 预期关注 |
|---|---|---|
| 基准组合 | 验证普通用户在 Web 渠道使用默认配置提交成功 | 主链路结果正确 |
| 高风险组合 | 验证管理员在灰度渠道使用新配置处理成功 | 配置生效且权限正确 |
| 非法组合 | 验证不支持的渠道与配置组合无法启用 | 系统阻止配置并提示原因 |
