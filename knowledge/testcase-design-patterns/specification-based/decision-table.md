# 判定表测试设计 DTT

## 方法定义

判定表测试设计用于描述“多个条件组合共同决定动作或结果”的业务规则。它把条件列为条件桩，把动作或结果列为动作桩，再将每一种有效规则组合转化为测试用例。

DTT 适合业务规则清晰但组合较多的场景，能防止遗漏互斥规则、缺省规则和冲突规则。

## 适用场景

- 多个条件共同决定是否允许、是否计费、是否审批、是否拦截。
- 存在业务规则矩阵、准入矩阵、费用矩阵、权限矩阵。
- 每个条件的取值有限，结果动作明确。
- 规则中存在优先级、互斥、缺省或例外。

## 不适用场景

- 只有单个判断点，应使用 DPT。
- 重点是状态从 A 到 B 的迁移，应使用 STT。
- 重点是周期边界和重复处理，应使用 PCT。
- 条件取值很多且结果相同，可先用 DCoT 裁剪。

## 建模输入

| 元素 | 说明 |
|---|---|
| 条件桩 | 影响结果的条件，例如角色、状态、金额、开关 |
| 条件项 | 条件的具体取值，例如是/否、通过/拒绝 |
| 动作桩 | 系统要执行的动作，例如允许、拒绝、计费、通知 |
| 规则列 | 一组条件项到动作项的映射 |
| 约束 | 互斥、不可能、优先级、缺省 |

## 设计步骤

1. 列出所有条件桩，只保留会影响结果的条件。
2. 列出动作桩，动作必须可观察或可验证。
3. 枚举条件组合。
4. 删除业务不可能组合，并记录原因。
5. 合并动作完全相同且风险较低的等价规则。
6. 标记冲突规则和缺省规则。
7. 每条有效规则至少派生一条用例。
8. 高风险规则补充异常输入、边界值或审计验证。

## 用例派生规则

| 规则类型 | 派生方式 |
|---|---|
| 正常允许规则 | 生成成功路径用例 |
| 正常拒绝规则 | 生成拦截或拒绝用例 |
| 缺省规则 | 生成未命中其他规则时的默认处理用例 |
| 冲突规则 | 生成优先级或冲突处理用例 |
| 不可能组合 | 不生成执行用例，在设计方案中说明 |

## 数量控制

- 条件数量小于等于 3 且风险高时，可覆盖全部有效规则。
- 条件数量较多时，优先覆盖不同动作结果和高风险规则。
- 同动作、同风险、同路径规则可合并为一条代表用例。
- `Level 0` 规则不得只用摘要覆盖，必须有可执行用例。

## 前置步骤关注

- 准备能够满足该规则列条件的数据状态。
- 明确用户角色、业务状态、配置开关和输入值。
- 若条件无法从界面设置，需要说明后台或数据准备方式。

## 测试步骤关注

- 按规则列逐步构造条件。
- 触发业务判定动作。
- 不在同一条用例中混入多个规则列。

## 预期结果关注

- 命中的规则正确。
- 执行动作正确，例如允许、拒绝、计费、通知、流转。
- 拒绝原因或提示明确。
- 审计、日志或状态变更符合规则。

## 常见误区

- 条件列不完整，遗漏缺省规则。
- 不删除不可能组合，导致生成不可执行用例。
- 把单个判断点硬扩展成大判定表，造成冗余。
- 只关注允许规则，遗漏拒绝和冲突处理。

## 设计方案片段

| 覆盖意图 | 用例标题建议 | 预期关注 |
|---|---|---|
| 允许规则 | 验证满足全部准入条件时审批提交成功 | 系统允许提交并进入下一状态 |
| 拒绝规则 | 验证用户角色不满足时审批提交被拒绝 | 提示无权限且状态不变 |
| 缺省规则 | 验证未命中特殊规则时按默认规则处理 | 命中默认动作并记录处理结果 |
