# 因果图测试设计

## 方法定义

因果图测试设计用于分析输入原因和输出结果之间的逻辑关系。它先识别原因、结果和约束，再将逻辑关系转换为判定表或关键组合用例。

因果图本身通常不是最终用例，而是建模工具。最终用例应落到判定表、数据组合或具体场景。

## 适用场景

- 多个输入原因通过与、或、非关系共同影响结果。
- 规则文本复杂，直接写判定表容易遗漏。
- 存在互斥、唯一、包含、要求等约束。
- 需要解释为什么某些条件组合会触发某个结果。

## 不适用场景

- 条件和动作关系已经清晰成表，应直接使用判定表。
- 只有单个判断点，应使用判定点。
- 重点是状态或周期，不应强行建立因果图。

## 建模输入

| 元素 | 说明 |
|---|---|
| 原因 | 输入条件、业务条件、环境条件 |
| 结果 | 系统动作、输出、状态变化、提示 |
| 逻辑关系 | 与、或、非、异或、蕴含 |
| 约束 | 互斥、唯一、包含、要求、屏蔽 |
| 派生表 | 从因果图转换出的判定表或组合表 |

## 设计步骤

1. 从测试点中抽取原因和结果。
2. 将复杂语句拆成原子原因。
3. 建立原因到结果的逻辑关系。
4. 标记约束：
   - 互斥：两个原因不能同时为真。
   - 唯一：多个原因中只能有一个为真。
   - 包含：一个原因为真时另一个原因必须为真。
   - 要求：某结果发生需要指定原因。
5. 将因果图转换为判定表或关键组合。
6. 删除不可能组合。
7. 为每个结果和关键约束派生用例。

## 用例派生规则

| 关系类型 | 派生方式 |
|---|---|
| 与关系 | 所有必要原因同时满足时触发结果 |
| 或关系 | 任一原因满足时触发结果 |
| 非关系 | 原因不满足时触发替代结果 |
| 互斥约束 | 验证互斥原因无法同时成立或按优先级处理 |
| 包含约束 | 验证缺少被包含条件时系统拦截 |

## 数量控制

- 因果图复杂时，最终用例不要覆盖所有布尔组合。
- 优先覆盖每个结果至少一次。
- 优先覆盖约束违反场景。
- 高风险结果保留多个关键原因组合。

## 前置步骤关注

- 准备能够分别满足关键原因的输入或业务状态。
- 对互斥或包含关系，明确如何构造违反约束的数据。
- 确认结果可观察。

## 测试步骤关注

- 按原因组合构造输入。
- 触发系统判断。
- 不在同条用例中验证多个结果。

## 预期结果关注

- 触发的结果与逻辑关系一致。
- 约束冲突被正确处理。
- 优先级、拒绝原因和状态变化可判定。
- 不可能组合不会被错误接受。

## 常见误区

- 把因果图作为最终输出，不转换为可执行用例。
- 原因不是原子条件，导致组合含糊。
- 忽略互斥和包含约束。
- 穷举所有组合，产生大量低价值用例。

## 设计方案片段

| 覆盖意图 | 用例标题建议 | 预期关注 |
|---|---|---|
| 与关系 | 验证满足所有准入原因时系统允许提交 | 触发允许结果 |
| 或关系 | 验证任一豁免条件满足时系统免审通过 | 命中豁免结果 |
| 约束违反 | 验证互斥条件同时出现时系统拒绝处理 | 提示条件冲突 |
