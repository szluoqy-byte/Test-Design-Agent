# 场景 / 用例 / 用户故事测试设计

## 方法定义

场景测试从用户目标和业务流程出发，设计端到端或局部业务链路用例。用例测试关注角色与系统之间的交互，用户故事测试关注验收标准是否被满足。

该模式适合把“主流程、备选流程、验收标准”转换为可执行步骤。

## 适用场景

- 测试点描述用户完成某个业务目标。
- 存在主流程、备选流程、异常流程。
- 需求以用户故事或验收标准表达。
- 需要验证多个页面、多个接口或多个业务对象协同。
- 测试点强调端到端链路。

## 不适用场景

- 单字段范围或格式校验，应使用等价类与边界值。
- 多条件规则矩阵，应使用判定表。
- 明确状态机，应使用状态转换。
- 周期性任务，应使用处理周期。

## 建模输入

| 元素 | 说明 |
|---|---|
| 角色 | 执行业务目标的用户或系统 |
| 目标 | 用户希望完成的业务结果 |
| 入口 | 页面、菜单、接口或任务入口 |
| 主成功路径 | 最常见的成功流程 |
| 备选路径 | 可替代但仍成功的流程 |
| 异常路径 | 失败、中断、取消、回退 |
| 结束状态 | 业务对象最终状态和可观察结果 |

## 设计步骤

1. 明确角色和业务目标。
2. 抽取主成功路径，形成主路径用例。
3. 抽取备选路径，形成必要的补充用例。
4. 识别异常、中断、取消、回退或失败恢复点。
5. 明确每条路径的结束状态和可观察结果。
6. 对核心链路补充数据一致性、通知、日志或审计验证。
7. 若缺少具体入口或菜单路径，使用中性入口表达并登记待确认。

## 用例派生规则

| 路径类型 | 派生方式 |
|---|---|
| 主路径 | 完成核心业务目标的成功用例 |
| 备选路径 | 使用替代入口、替代数据或替代操作完成目标 |
| 异常路径 | 中途失败、校验不通过或依赖不可用 |
| 中断恢复 | 取消、返回、刷新、重新进入后的处理 |
| 端到端链路 | 从入口到最终状态的完整业务验证 |

## 数量控制

- `Level 0` 场景至少包含主路径和关键异常。
- 备选路径只有在真实存在业务价值时生成。
- 同一主路径下的字段边界不要全部塞进场景用例，交给边界模式。
- 端到端用例不宜过长，超过一个独立目标应拆分。

## 前置步骤关注

- 准备角色、业务对象、基础数据和系统状态。
- 明确流程入口；缺少入口时使用“进入相关业务入口”并待确认。
- 准备满足主路径或异常路径的数据。

## 测试步骤关注

- 采用用户真实操作顺序。
- 每步只描述一个明确动作。
- 跨页面或跨系统时说明跳转或触发关系。
- 不把多个备选路径写入一条主路径用例。

## 预期结果关注

- 用户目标是否达成。
- 业务对象状态是否正确。
- 页面展示、消息通知、记录和数据一致性是否正确。
- 异常路径是否给出明确反馈并保持安全状态。

## 常见误区

- 把场景用例写成“验证功能正常”。
- 一条用例覆盖多个互相独立的用户目标。
- 主路径缺少结束状态，只写“提交成功”。
- 未区分备选路径和异常路径。

## 设计方案片段

| 覆盖意图 | 用例标题建议 | 预期关注 |
|---|---|---|
| 主路径 | 验证普通用户完成订单提交主流程 | 订单生成且状态正确 |
| 备选路径 | 验证用户使用备用地址完成订单提交 | 地址生效且订单生成 |
| 异常路径 | 验证用户取消提交后订单不被创建 | 系统返回原页面且无新订单 |
