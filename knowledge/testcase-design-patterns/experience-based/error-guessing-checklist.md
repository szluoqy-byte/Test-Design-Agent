# 基于经验的错误推测与检查清单

## 方法定义

基于经验的测试通过历史缺陷、专家经验、上线事故、常见遗漏和系统特性推测潜在错误，并补充规格测试可能遗漏的用例。它不是替代基于规格的测试，而是在已有设计方案上增加高价值风险场景。

错误推测强调“哪里最可能出错”，检查清单强调“哪些常见问题必须扫一遍”。

## 适用场景

- 测试点来自历史缺陷或线上事故。
- 模块存在长期高发问题。
- 需求描述较简单，但测试人员知道实现易错点。
- 涉及缓存、异步、权限、并发、幂等、数据一致性。
- 评审时发现规格模式覆盖了规则，但没有覆盖常见失败方式。

## 不适用场景

- 需求规则尚不清楚时，不能只靠经验猜测生成用例。
- 已有明确范围、状态、判定、周期或接口结构时，应先使用对应规格模式。
- 经验项没有可观察结果时，不应强行生成用例。

## 输入来源

| 来源 | 示例 |
|---|---|
| memory | 历史缺陷、评审偏好、常见遗漏 |
| 测试点风险备注 | “历史出现过重复扣款”“容易越权” |
| 业务域特点 | 资金、库存、缓存、异步任务 |
| 技术实现特点 | 队列、定时任务、分布式锁、回调 |
| 专家检查清单 | 空值、重复提交、并发、超时 |

## 设计步骤

1. 读取测试点和 context pack，识别是否存在历史风险。
2. 判断风险是否已被主规格模式覆盖。
3. 未覆盖时生成补充用例；已覆盖时在原用例中强化预期。
4. 将经验项转化为可执行触发条件。
5. 明确可观察结果，例如状态、记录、日志、通知、拦截。
6. 根据风险影响调整用例等级。
7. 去除无法触发、无法观察或与现有用例重复的经验项。

## 常见检查清单

| 类别 | 检查项 | 常见用例方向 |
|---|---|---|
| 空值与缺省 | 必填为空、选填为空、默认值未配置 | 拦截、默认处理 |
| 重复提交 | 连续点击、重复请求、重复回调 | 幂等、防重复 |
| 并发 | 多用户同时操作同一对象 | 锁、库存、状态一致 |
| 权限 | 越权访问、角色切换、租户隔离 | 拦截、数据隔离 |
| 缓存 | 修改后缓存未刷新、旧数据展示 | 前后端一致 |
| 异步 | 任务延迟、回调失败、消息重复 | 重试、补偿、最终一致 |
| 边界 | 时间跨天、金额精度、数量为 0 | 边界处理 |
| 审计 | 关键操作无日志、日志字段缺失 | 可追踪 |

## 用例派生规则

| 经验信号 | 派生方式 |
|---|---|
| 历史缺陷 | 生成同类回归用例，并明确“不复现”的可观察结果 |
| 常见遗漏 | 生成检查清单补充用例 |
| 高发风险 | 提高等级或补充异常路径 |
| 技术风险 | 叠加幂等、重试、并发、缓存一致性验证 |

## 数量控制

- 每个测试点最多补充 1 到 3 个经验用例。
- 与主模式完全重复的经验项不单独生成。
- 低风险经验项可写入覆盖审查备注，不生成用例。
- 历史线上事故和资金类缺陷优先保留独立用例。

## 前置步骤关注

- 构造历史缺陷复现条件或同类条件。
- 准备并发、重复、缓存、异步所需的数据状态。
- 明确如何观察是否复现。

## 测试步骤关注

- 操作要能触发易错点，例如重复点击、重复请求、切换角色、等待任务。
- 不要只写“检查历史问题是否修复”，必须写出触发动作。
- 并发和异步用例要说明触发顺序或等待条件。

## 预期结果关注

- 历史缺陷不复现。
- 重复、并发或异步场景下数据一致。
- 越权或非法操作被拦截。
- 缓存、日志、通知、审计符合预期。

## 常见误区

- 把经验清单当成固定模板，所有模块都套一遍。
- 只写“验证历史缺陷已修复”，没有执行步骤。
- 经验用例和规格用例重复，造成冗余。
- 缺少可观察结果，导致无法判定。

## 设计方案片段

| 经验信号 | 用例标题建议 | 预期关注 |
|---|---|---|
| 重复提交 | 验证用户连续提交订单时系统只生成一笔订单 | 结果幂等，不重复生成 |
| 缓存风险 | 验证权限变更后用户立即访问旧入口被拦截 | 缓存刷新且权限生效 |
| 历史缺陷 | 验证回调重复到达时结算记录不重复生成 | 只保留一条有效记录 |
