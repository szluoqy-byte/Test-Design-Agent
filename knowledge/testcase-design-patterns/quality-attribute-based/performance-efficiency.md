# 基于质量属性的性能效率测试设计

## 方法定义

性能效率测试设计关注系统在指定负载、数据量或资源约束下完成业务处理的能力。它验证响应时间、吞吐量、容量、资源使用和批处理耗时是否满足预期。

该模式不替代性能测试工具或压测脚本，而是指导从测试点生成可执行的性能效率验证用例和可观察预期。

## 适用场景

- 测试点提到性能、响应时间、吞吐量、容量、并发、批量处理。
- 大数据量列表、导出、统计、搜索、分页。
- 定时任务或批处理耗时。
- 接口在典型负载下的响应表现。
- 用户操作要求在可接受时间内完成。

## 不适用场景

- 只验证功能是否正确，应使用基于规格的测试。
- 重点是失败恢复、重试、降级，应使用可靠性与恢复性。
- 没有任何可观察指标时，应先澄清指标或使用保守阈值表达。

## 建模输入

| 元素 | 说明 |
|---|---|
| 业务场景 | 被验证的操作、接口或任务 |
| 性能指标 | 响应时间、吞吐量、耗时、资源占用 |
| 负载条件 | 用户数、请求数、数据量、批量大小 |
| 观察方式 | 页面耗时、接口耗时、任务日志、监控指标 |
| 通过阈值 | 明确阈值或项目约定 |
| 数据规模 | 基准、典型、大数据量、峰值 |

## 设计步骤

1. 明确业务动作和性能指标。
2. 明确测试负载：单用户、典型并发、峰值、批量数据。
3. 明确测试数据规模和环境前提。
4. 设计基准性能用例。
5. 设计容量或大数据量用例。
6. 对高风险场景补充峰值、连续执行或资源使用验证。
7. 如果没有明确阈值，写入待确认问题，并使用“符合项目性能基线”作为临时预期。

## 用例派生规则

| 场景类型 | 派生方式 |
|---|---|
| 基准响应 | 单用户或低负载下完成业务操作 |
| 典型负载 | 常规并发或常规数据量 |
| 容量边界 | 接近业务上限的数据量或请求量 |
| 峰值负载 | 高峰并发或批量处理 |
| 资源效率 | CPU、内存、数据库连接、队列积压 |

## 数量控制

- 没有明确性能目标时，不扩展大量性能用例。
- `Level 0/1` 性能测试点至少覆盖基准和关键容量边界。
- 普通性能关注点可只生成一条典型负载用例。
- 性能用例应避免与功能用例重复，只强调负载和指标。

## 前置步骤关注

- 准备指定规模的数据量。
- 明确测试环境、账号、配置和监控入口。
- 明确性能基线或阈值来源。
- 准备清理或隔离历史数据，避免影响结果。

## 测试步骤关注

- 触发业务操作并记录开始、结束或监控指标。
- 对并发或批量场景说明负载规模。
- 对任务耗时说明任务触发和完成观察方式。

## 预期结果关注

- 响应时间或任务耗时满足阈值。
- 吞吐量达到预期。
- 大数据量下结果正确且页面或接口可用。
- 资源使用无异常飙升。
- 超时或接近阈值时有合理提示或保护。

## 常见误区

- 只写“性能正常”，没有指标。
- 不说明数据量、并发量或负载条件。
- 把性能用例写成普通功能用例。
- 没有观察方式，导致无法判定。

## 设计方案片段

| 性能信号 | 用例标题建议 | 预期关注 |
|---|---|---|
| 响应时间 | 验证订单列表在典型数据量下查询响应满足基线 | 页面在阈值内返回结果 |
| 容量 | 验证导出最大允许数据量时任务可完成 | 导出任务完成且耗时满足要求 |
| 批处理 | 验证结算任务处理典型批量数据的耗时 | 任务日志显示耗时达标 |
