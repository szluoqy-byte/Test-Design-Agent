# 基于质量属性的可靠性与恢复性测试设计

## 方法定义

可靠性与恢复性测试设计关注系统在故障、超时、重复、依赖不可用或异常中断情况下，是否能保持稳定、避免数据损坏，并通过重试、降级、补偿或恢复机制回到可接受状态。

该模式重点不是“功能成功”，而是“失败时是否安全，恢复后是否一致”。

## 适用场景

- 依赖系统失败、接口超时、消息丢失或重复。
- 重试、降级、熔断、补偿、断点续跑。
- 异步任务、回调、定时任务、批处理。
- 数据最终一致性、失败恢复、人工补偿。
- 高风险链路要求故障可追踪、可恢复。

## 不适用场景

- 只验证字段规则或主流程成功，应使用基于规格的测试。
- 只关注响应时间和吞吐量，应使用性能效率。
- 未定义任何故障点或恢复机制时，应登记待确认问题。

## 建模输入

| 元素 | 说明 |
|---|---|
| 故障点 | 依赖失败、超时、重复、异常中断 |
| 保护机制 | 重试、降级、补偿、回滚、人工处理 |
| 恢复目标 | 最终状态、数据一致、任务继续 |
| 观察对象 | 状态、日志、告警、补偿记录、消息 |
| 重试策略 | 次数、间隔、停止条件 |
| 安全状态 | 失败时允许保持的状态 |

## 设计步骤

1. 识别业务链路中的故障点。
2. 明确故障发生时系统应如何保护数据。
3. 明确恢复机制和恢复目标。
4. 设计依赖失败、超时、重复请求或重复消息用例。
5. 设计重试、降级、补偿、恢复后继续处理用例。
6. 验证最终状态、日志、告警和审计记录。
7. 对关键状态、数量扣减、资源占用类场景补充幂等和最终一致性验证。

## 用例派生规则

| 故障类型 | 派生方式 |
|---|---|
| 依赖失败 | 依赖不可用时系统降级、失败提示或保持安全状态 |
| 超时 | 超时后重试、失败返回或任务挂起 |
| 空数据 | 返回空集合或无可处理数据时保持可用并给出可理解状态 |
| 异常数据 | 字段缺失、格式异常或部分数据不可用时降级展示、跳过或失败提示 |
| 重复 | 重复请求、重复消息、重复回调保持幂等 |
| 中断 | 任务中断后可恢复或可人工补偿 |
| 补偿 | 失败后执行补偿并记录结果 |
| 恢复 | 故障恢复后继续处理积压或失败数据 |

## 数量控制

- `Level 0/1` 可靠性测试点至少覆盖关键故障和恢复。
- 普通可靠性关注点可选择一个代表性故障点。
- 同类依赖失败可合并，保留影响最大的依赖。
- 如果恢复机制未明确，应生成待确认问题，不编造。

## 前置步骤关注

- 准备可触发故障的依赖、消息、任务或数据。
- 明确如何模拟失败、超时、重复或中断。
- 准备观察日志、告警、补偿记录和最终状态的入口。

## 测试步骤关注

- 先触发正常业务动作，再制造故障或重复。
- 对恢复场景，应先失败，再恢复依赖并触发继续处理。
- 对重复场景，应重复同一业务请求或消息。

## 预期结果关注

- 失败时系统保持安全状态。
- 不产生重复处理、重复资源扣减、重复记录或重复状态变化。
- 重试次数、间隔和终止条件符合规则。
- 降级结果可识别，用户或调用方获得明确反馈。
- 补偿记录、日志和告警完整。
- 恢复后最终数据一致。

## 常见误区

- 只验证失败提示，不验证数据是否安全。
- 不验证恢复后的最终一致性。
- 把重试和重复请求混为一谈。
- 未说明如何模拟故障，导致用例无法执行。
- 只覆盖服务失败或超时，遗漏空数据、部分字段缺失、返回格式异常和故障解除后的恢复验证。

## 设计方案片段

| 可靠性信号 | 用例标题建议 | 预期关注 |
|---|---|---|
| 依赖失败 | 验证外部依赖不可用时业务对象保持安全状态 | 不产生错误状态变化，并记录失败原因 |
| 重复消息 | 验证重复回调到达时系统保持幂等 | 只处理一次且记录重复回调 |
| 恢复补偿 | 验证处理任务失败恢复后可继续处理 | 失败数据最终处理完成且可追踪 |
