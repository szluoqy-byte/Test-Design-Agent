# 测试用例审查

本 skill 负责在输出前审查测试用例质量，并生成覆盖审查和质量门禁结果。审查重点是测试步骤和 GUI 自动化可执行性，覆盖、等级、去重和专家评分只做轻量一致性检查。

## 路径解析规则

- 本 skill 中所有 `knowledge/...`、`templates/...`、`quality-gates/...` 路径均为仓库根目录相对路径。
- 不要把知识、模板或门禁文件解析到 `skills/testcase-review/` 子目录下。

## 职责边界

- 负责“是否达标”的审查和问题定位，不重新生成完整设计方案，也不大段重写测试用例明细。
- 优先检查测试步骤是否状态独立、是否第 1 步登录带环境类型标签的环境、是否包含完整入口链路、GUI 动作是否具备可自动化定位对象。
- 用例明细在进入本 skill 前应已通过 `bin/lint-testcase-report.py` 和 `bin/semantic-testcase-check.py` 的确定性预检；如未通过，优先回到 `testcase-writing` 定向修正，不进入专家评审。
- 可提出修正建议；明显结构错误应输出精确问题清单，并要求对应阶段修正后重跑门禁。
- 信息缺失导致无法修正时，输出待确认问题。
- 不要求单条用例明细写入测试点 ID，追溯只看追溯矩阵。

## 输入

- 测试用例设计报告草稿。
- 独立测试用例明细草稿。
- 规范化测试点清单。
- `context-pack.md`。

## 审查步骤

1. 按仓库根目录下的 `quality-gates/testcase-schema-check.md` 检查用例字段。
2. 按仓库根目录下的 `quality-gates/testcase-executability-check.md` 重点检查测试步骤和 GUI 自动化可执行性。
3. 按仓库根目录下的 `quality-gates/expected-result-check.md` 检查预期是否可观察、可判定、可对应断言。
4. 按仓库根目录下的 `knowledge/coverage-traceability-standard.md` 轻量检查追溯矩阵是否完整、编号是否可解析。
5. 按仓库根目录下的 `quality-gates/coverage-check.md` 只检查明显未覆盖或高风险漏测，不做扩展性深挖。
6. 按仓库根目录下的 `quality-gates/level-consistency-check.md` 只检查明显等级错误，不因等级建议触发重写。
7. 按仓库根目录下的 `quality-gates/duplicate-testcase-check.md` 只检查明显重复，不因表述相近但覆盖目标不同而合并。
8. 按仓库根目录下的 `quality-gates/expert-review-rubric.md` 给出简要专家评分，评分不阻断交付。

## 输出

- 覆盖审查结果。
- 质量门禁结果。
- 专家评审评分。
- 阻断项、影响范围、修正归属和建议修正。

## 规则

- 输出质量问题时先定位到测试点、用例编号、字段或报告章节，再给出修正建议；不要用整体重写替代问题定位。
- 确定性格式、编号、有序列表、步骤/预期数量、GUI 登录环境类型标签、完整入口链路、GUI 操作对象、空泛描述和句末标点问题，归属 `testcase-writing` 定向修正。
- 追溯引用、覆盖状态、等级一致性和重复用例问题，归属 `testcase-review` 生成轻量修正建议；除追溯缺失、用例不可执行、明显漏测外，不作为阻断项。
- 专家评分只在确定性脚本和基础质量门禁通过后执行；评分不触发大段重写，只输出扣分原因和必要的最小修正建议。
- 输出质量问题优先形成修正闭环，不直接把明显错误留给用户确认。
- 信息不足导致无法可靠修正时，生成待确认问题。
- 追溯关系只在追溯矩阵中维护，不要求单条用例明细写测试点 ID。
- 质量门禁结果表的门禁名称固定使用：`结构字段`、`可执行性`、`预期可判定`、`追溯覆盖`、`覆盖完整性`、`等级一致性`、`重复用例`；结果只使用 `通过`、`未通过`、`待确认`、`不适用`。
