# 待确认治理

本 skill 只管理待确认问题，不负责生成测试用例，也不在流程中打断用户。所有需要用户确认的信息统一沉淀到最终报告的“待确认问题”章节。

## 职责边界

- 只管理待确认候选、分级、去重和记录决策。
- 不补写业务事实。
- 不直接生成测试用例。
- 已被用户回答、memory 覆盖或重复的问题必须删除。
- 不向用户提问，不要求暂停，不决定流程是否继续。

## 输入

- 待确认检查点：`CP-INPUT`、`CP-DESIGN`、`CP-WRITING`、`CP-REVIEW`。
- 候选问题清单。
- 当前测试点、context pack 和已确认回答。

## 分级

| 等级 | 触发条件 | 默认处理 |
|---|---|---|
| MustConfirm | 缺口会导致用例不可执行、预期无法判定或无法可靠覆盖测试点 | 写入最终报告待确认问题；对应测试点、覆盖状态或质量门禁标记为 `待确认`，流程继续 |
| ShouldConfirm | 缺口影响高风险用例质量，但可先生成保守用例 | 写入最终报告待确认问题，流程继续 |
| RecordOnly | 可用中性占位表达，不影响结构 | 写入待确认问题 |
| Ignore | 重复、低价值或已被上下文覆盖 | 删除 |

## 输出

| 问题 ID | 检查点 | 等级 | 影响范围 | 建议确认事项 | 当前处理 |
|---|---|---|---|---|---|

## 规则

- 已回答问题不得重复记录。
- 同类问题合并为一个问题。
- 单个检查点最多保留 1 到 3 个高价值待确认问题，优先保留会影响可执行性、预期判定、覆盖状态或 Level 0/Level 1 核心用例质量的问题。
- 批量生成测试用例时，任何等级都不得中断当前流程；所有未解决问题都进入最终报告“待确认问题”并继续执行。
- 缺少具体账号、菜单、按钮、测试数据、配置或提示文案时，如可使用中性可替换表达且不影响用例结构和基本判定，应归为 `RecordOnly` 或 `ShouldConfirm`。
- 用户回答默认只作用于本次运行；未经确认不写入 memory。
- 最终报告只保留未解决问题。
