# 测试用例设计

本 skill 负责从测试点生成用例设计方案。粒度边界读取仓库根目录下的 `knowledge/test-scenario-point-case-boundary.md`，测试类型识别读取仓库根目录下的 `knowledge/basic-test-types.md`，详细设计模式读取仓库根目录下的 `knowledge/testcase-design-patterns/README.md` 和对应模式文件。

## 路径解析规则

- 本 skill 中所有 `knowledge/...`、`templates/...`、`quality-gates/...` 路径均为仓库根目录相对路径。
- 不要把知识文件解析到 `skills/testcase-design/knowledge/` 或其他 skill 子目录下。
- 读取测试设计模式时，应先读取仓库根目录下的 `knowledge/testcase-design-patterns/README.md`，再按 README 路由读取同目录下的具体模式文件。

## 职责边界

- 输出“如何设计”的方案，不输出完整用例正文。
- 选择测试设计模式并说明覆盖意图。
- 为 `testcase-writing` 提供用例标题、等级、前置关注、步骤关注和预期关注。
- 不编写最终的前置步骤、测试步骤和预期结果。

## 输入

- 规范化测试点清单。
- `context-pack.md` 中与本次相关的项目事实。
- 用户补充的本次设计约束。

## 设计步骤

1. 按仓库根目录下的 `knowledge/test-scenario-point-case-boundary.md` 检查测试点是否写成测试场景或测试用例，必要时提出拆分、上收或待确认建议。
2. 识别每条测试点的测试类型和主要信号：流程、边界、组合、判定、周期、状态、接口、历史缺陷、风险或质量属性。
3. 按仓库根目录下的 `knowledge/testcase-design-patterns/README.md` 选择一个主模式，必要时选择一个辅助模式。
4. 确定覆盖意图：主路径、异常路径、边界、非法迁移、周期边界、接口契约、风险防护、性能效率或可靠性恢复。
5. 对常见遗漏信号执行覆盖补全判断：
   - 选择、下拉、枚举类信号：判断是否需要覆盖选项展示、选择、清除、收起、选中态保持和选项来源异常。
   - 查询筛选类信号：判断是否需要覆盖单条件、组合条件、无匹配、条件保持和结果与条件一致。
   - 多条件组合信号：判断是否需要覆盖全部条件满足、仅部分条件满足、全部条件不满足。
   - 列表分页或容量信号：判断是否需要覆盖小于、等于、大于容量边界，以及翻页后的条件保持。
   - 异常或可靠性信号：判断是否需要覆盖失败、超时、空数据、字段缺失、格式异常和故障恢复后再次操作。
   - 新增入口、页签或功能改动信号：判断是否需要覆盖相邻既有功能或继承功能影响。
6. 按用例等级定位控制覆盖深度和用例数量：
   - `Level 0`：核心用例，聚焦每个转测试版本必验的核心功能；执行失败应视为版本核心功能受损。
   - `Level 1`：关键用例，覆盖特性关键功能、关键可靠性；建议每个迭代验证。
   - `Level 2`：重要用例，覆盖系统重要功能；建议 TR 点或对外发布版本完整验证。
   - `Level 3`：一般用例，用于版本变更较大或需要全量验证时的回归覆盖。
   - `Level 4`：生僻用例，覆盖低频应用场景和特殊预置条件，建议新特性首次验证后按需回归。
7. 判断是否存在同一场景、同一执行链路下可由一条用例自然覆盖的多个测试点；如存在，保留主覆盖测试点，并在设计意图或待确认信息中说明其他可被追溯覆盖的测试点。
8. 输出设计方案表，供 `testcase-writing` 编写用例。

## 输出格式

| 测试点 ID | 测试点摘要 | 主设计模式 | 辅助模式 | 覆盖意图 | 建议用例数 | 默认等级 | 待确认信息 |
|---|---|---|---|---|---|---|---|

## 约束

- 不把“正向/异常”作为独立设计模式；它们是每种模式下的路径属性。
- 不把权限、数据一致性等业务关注点单独当作模式；应映射到判定表、判定点、场景、接口契约、状态转换、处理周期或风险设计。
- 不为了凑数量生成重复用例。
- 不强制每个测试点生成独立用例；如果多个测试点属于同一场景、共享同一执行链路且不是独立分支、异常或风险，可以由一条用例覆盖，并在追溯矩阵中表达。
- 如果覆盖补全判断依赖的枚举值、容量阈值、兼容范围、性能指标或异常处理规则缺失，应登记待确认信息，不编造具体取值。
