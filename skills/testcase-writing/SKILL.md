# 测试用例编写

本 skill 负责把设计方案写成可执行测试用例。字段和写法遵循 `knowledge/testcase-standard.md`、`knowledge/testcase-writing-guide.md` 和 `templates/testcase-detail-template.md`。

## 职责边界

- 负责“写成可执行用例”，不重新选择测试设计模式。
- 只输出单条用例明细的 6 个必需字段。
- 测试点关联、覆盖状态、质量结论交给 `testcase-review`。
- 信息不足时使用中性可替换表达，并把缺口交给待确认问题。

## 输入

- 用例设计方案。
- 规范化测试点清单。
- 本次 `context-pack.md`。
- 已确认的澄清结果。

## 输出字段

每条测试用例只输出以下字段：

- 用例编号
- 用例名称
- 用例等级
- 前置步骤
- 测试步骤
- 预期结果

## 编写规则

1. 用例编号使用 `TC-001` 起的连续编号，除非项目 memory 明确要求其他编号规则。
2. 用例名称使用“验证 + 被测对象 + 场景 + 预期目标”的结构。
3. 用例等级使用 `Level 0` 至 `Level 4`。
4. 前置步骤写执行前必须满足的账号、数据、环境、配置和状态。
5. 测试步骤使用有序列表，操作对象明确，避免“验证功能正常”“检查结果正确”等空泛描述。
6. 预期结果必须可观察、可判定，并与测试步骤逐条或分组对应。
7. 不知道具体入口、按钮、账号或数据时，使用中性可替换表达，并把缺口交给待确认问题。

## 禁止项

- 不在单条用例明细中输出关联测试点、模块、用例类型、测试数据、自动化建议或风险备注。
- 不把多个独立验证目标合并成一条过长用例。
- 不编造未输入、未在 memory 中确认的业务事实。
