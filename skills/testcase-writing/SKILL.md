# 测试用例编写

本 skill 负责把设计方案写成可执行测试用例。字段和写法遵循仓库根目录下的 `knowledge/testcase-standard.md`、`knowledge/testcase-writing-guide.md` 和 `templates/testcase-detail-template.md`。

## 路径解析规则

- 本 skill 中所有 `knowledge/...`、`templates/...`、`quality-gates/...` 路径均为仓库根目录相对路径。
- 不要把知识、模板或门禁文件解析到 `skills/testcase-writing/` 子目录下。

## 职责边界

- 负责“写成可执行用例”，不重新选择测试设计模式。
- 只输出单条用例明细的 6 个必需字段。
- 测试点关联、覆盖状态、质量结论交给 `testcase-review`。
- 信息不足时使用中性可替换表达，并把缺口交给待确认问题。

## 输入

- 用例设计方案。
- 规范化测试点清单。
- 本次 `context-pack.md`。
- 已确认的澄清结果。

## 输出字段

每条测试用例只输出以下字段：

- 用例编号
- 用例名称
- 用例等级
- 前置步骤
- 测试步骤
- 预期结果

## 编写规则

1. 用例编号使用 `TC-001` 起的连续编号，除非项目 memory 明确要求其他编号规则。
2. 用例名称使用“验证 + 被测对象 + 场景 + 预期目标”的结构。
3. 用例等级使用 `Level 0` 至 `Level 4`。
4. 前置步骤写执行前必须满足的账号、数据、环境、配置和状态，优先使用输入中“前置条件”“执行用户/角色”“测试数据因子”“业务设计约束”和“补充说明”提供的信息。
5. 测试步骤使用有序列表，操作对象明确，避免“验证功能正常”“检查结果正确”等空泛描述。
6. 每条 GUI 场景测试用例的测试步骤都应包含从已登录或约定起始页面到目标页面、页签或功能区的完整入口操作链路，不仅在前置步骤中写“已进入目标页面”。入口步骤应写成“点击【一级入口】 > 【二级入口】 > 【三级入口】 > 【目标页面/页签】进入目标页面”这类可执行路径。
7. 入口路径来源优先级为：当前输入中明确给出的完整场景入口/触发方式 > context-pack 中基于 domain memory 补全的建议入口路径 > 中性可替换表达。当前输入只给出较短入口或目标页面/页签时，可使用 context-pack 补全父级菜单链路；当前输入与 context-pack 在同一层级冲突时，使用当前输入，并把冲突交给待确认问题。
8. 选择、下拉、枚举类用例必须把打开控件、选择选项、必要时清除选择或收起控件写成可执行步骤，不只写“查看下拉选项”。
9. 查询筛选类用例必须写明输入或选择条件、触发查询、观察结果字段或状态；组合筛选用例应体现全部满足、部分满足或全部不满足的覆盖意图。
10. 分页或容量类用例必须写明数据量条件、查询或加载动作、分页组件观察、翻页或页容量操作。
11. GUI 场景测试用例涉及页面、菜单、页签、表单、输入框、下拉框、按钮、弹窗、表格、分页等对象时，步骤必须写出操作对象、用户动作和必要输入；预期必须写出可观察的 UI 状态、文案、选中态、表格字段、结果范围、空态、分页状态或提示信息。
12. GUI 场景测试中，不使用“查看页面”“验证布局”“检查列表”“执行查询”等泛化步骤承载多个观察点；如需观察多个对象，应拆成筛选区、结果区、分页区、提示区等可对应预期的步骤。
13. 同一 GUI 控件或同一路径需要覆盖多个输入值时，步骤中必须显式列出输入值清单或选择项清单；不同输入值的预期差异明显、风险不同或失败定位方向不同，应拆成多条用例。
14. 接口测试用例不套用 GUI 控件规则，应围绕请求方式、路径、参数、字段、状态码、错误码、鉴权、幂等、性能或可靠性目标编写。
15. 故障类用例中，如果故障注入、Mock、断网、超时配置或服务开关是执行动作，应写入测试步骤；前置步骤只放执行前已具备的环境和数据条件。
16. 预期结果必须可观察、可判定，并与测试步骤逐条一一对应；测试步骤有 N 条，预期结果也必须有 N 条。
17. 前置步骤、测试步骤和预期结果都必须使用有序列表，且至少包含一条可执行或可观察内容；不要输出空字段或纯说明文本。
18. 测试步骤和预期结果的每一条都应使用完整句子，并以中文句号或英文句号结尾，避免触发格式类返工。
19. 禁止在用例明细中使用 `功能正常`、`流程正常`、`结果正确`、`符合预期`、`正常显示`、`正常处理`、`检查结果`、`进行测试` 等空泛短语；如确需表达成功或失败，必须同时说明可观察对象、状态或返回内容。
20. 每条预期结果至少包含明确观察对象或观察动作，例如页面展示、字段显示、按钮状态、接口响应、错误码、记录、日志、通知、权限拦截、状态保持等。
21. 输出前按 `bin/lint-testcase-report.py` 和 `bin/semantic-testcase-check.py` 的确定性规则自检，优先消除编号、字段、有序列表、步骤/预期数量、空泛描述、可观察对象和句末标点问题。
22. 不知道具体入口、按钮、账号、配置、数据准备方式或接口契约信息时，使用中性可替换表达，并把缺口交给待确认问题。

## 禁止项

- 不在单条用例明细中输出关联测试点、模块、用例类型、测试数据、自动化建议或风险备注。
- 不把多个独立验证目标合并成一条过长用例。
- 不编造未输入、未在 memory 中确认的业务事实。
